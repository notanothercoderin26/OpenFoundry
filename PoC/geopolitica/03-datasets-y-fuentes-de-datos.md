## 1️⃣ GDELT 2.0 — events and GKG (the workhorse)

**What it is:** the Global Database of Events, Language, and Tone. Two related feeds:
- **Event database**: structured rows with CAMEO event codes, actors (`Actor1Code`, `Actor1Name`, `Actor1Type1Code`, …), geolocation (`ActionGeo_Lat`, `ActionGeo_Long`, `ActionGeo_FullName`), and tone fields.
- **Global Knowledge Graph (GKG)**: per-article entries listing persons, organizations, locations, themes, and emotion scores mentioned in news articles worldwide.

**Why it's the workhorse:** the only fully-public, machine-readable, attribution-friendly source that gives us events + entities + locations at the volume needed to claim TB scale. CAMEO coding gives us a vocabulary for event types.

### A) Historical (batch) — CSV ZIPs every 15 min
- URL pattern (events): `http://data.gdeltproject.org/gdeltv2/YYYYMMDDHHMMSS.export.CSV.zip`
- URL pattern (GKG): `http://data.gdeltproject.org/gdeltv2/YYYYMMDDHHMMSS.gkg.csv.zip`
- A daily file list is at `http://data.gdeltproject.org/gdeltv2/lastupdate.txt` (rolling, 96 entries / day).

#### Bulk download script (12 months, single shard)
```bash
START=2025-04-01
END=2026-04-01
mkdir -p /data/raw/gdelt/events /data/raw/gdelt/gkg
python tools/poc-geopolitica/download_gdelt.py \
  --start "$START" --end "$END" \
  --feeds events,gkg \
  --output-events  /data/raw/gdelt/events/  \
  --output-gkg     /data/raw/gdelt/gkg/     \
  --partition-by day --convert-to parquet --workers 8

# Push to lake
aws s3 sync /data/raw/gdelt/ s3://acme-intel-poc/raw/gdelt/
```

### B) Live (streaming) — every 15 min during the demo
Implement a connector of type `gdelt-15min-poller` in `connector-management-service` that polls `lastupdate.txt`, downloads the new shards, and emits records into Kafka topics `gdelt.events.live` and `gdelt.gkg.live`.

#### Connector configuration (JSON for `connector-management-service`)
```json
{
  "name": "gdelt-live",
  "type": "gdelt-15min-poller",
  "manifest_url": "http://data.gdeltproject.org/gdeltv2/lastupdate.txt",
  "feeds": ["events", "gkg"],
  "poll_interval_seconds": 60,
  "sink": {
    "type": "kafka",
    "topic_events": "gdelt.events.live",
    "topic_gkg": "gdelt.gkg.live"
  }
}
```

### Mandatory attribution
> *"Source: GDELT Project — gdeltproject.org. CC-BY."* shown in the dashboard footer.

## 2️⃣ ACLED — armed conflict and protest events

**What it is:** the Armed Conflict Location & Event Data Project. Curated, geo-tagged conflict + protest events with high-quality typing (`event_type`, `sub_event_type`), actor names, fatalities, location.

**Why we use it:** ACLED corrects for GDELT's noise. We surface ACLED events as the **high-confidence** layer on top of the GDELT firehose.

**License caveat:** the academic license is free; commercial use (which a paid customer pilot qualifies as) requires a license. For the PoC we run under the academic/non-commercial license and **flag in the demo** that a commercial pilot requires renewing the ACLED contract.

### Download
```bash
# Account at https://acleddata.com — request API key
ACLED_KEY=...; ACLED_EMAIL=...
curl -fsSL \
  "https://api.acleddata.com/acled/read.csv?key=${ACLED_KEY}&email=${ACLED_EMAIL}&year=2024|2025|2026" \
  -o /data/raw/acled/2024-2026.csv
aws s3 cp /data/raw/acled/2024-2026.csv s3://acme-intel-poc/raw/acled/
```

### Mandatory attribution
> *"Source: ACLED — acleddata.com. Academic license."*

## 3️⃣ OFAC SDN — US Treasury sanctions list

**What it is:** the consolidated US sanctions list (Specially Designated Nationals + Blocked Persons + other programs).
- URL (XML): `https://www.treasury.gov/ofac/downloads/sdn.xml`
- URL (consolidated, with aliases / addresses): `https://www.treasury.gov/ofac/downloads/sdn_advanced.xml`

```bash
curl -fsSL https://www.treasury.gov/ofac/downloads/sdn_advanced.xml \
  -o /data/raw/ofac/sdn_advanced.xml
aws s3 cp /data/raw/ofac/sdn_advanced.xml s3://acme-intel-poc/raw/ofac/
```

A daily poller in `connector-management-service` re-fetches and lands a new transaction in the `raw.ofac_sdn` dataset.

## 4️⃣ EU Consolidated Financial Sanctions List

**What it is:** the EU's consolidated list of persons, groups, and entities subject to EU financial sanctions.
- URL (XML, public mirror via Commission FSF): an authenticated download token is required from the FSF service. For the PoC, use the public mirror documented at <https://webgate.ec.europa.eu/fsd/fsf/public/files/xmlFullSanctionsList_1_1/content?token=...>; the token is rotated, so the download script reads it from `EU_FSF_TOKEN`.

```bash
curl -fsSL "https://webgate.ec.europa.eu/fsd/fsf/public/files/xmlFullSanctionsList_1_1/content?token=${EU_FSF_TOKEN}" \
  -o /data/raw/eu_sanctions/full.xml
aws s3 cp /data/raw/eu_sanctions/full.xml s3://acme-intel-poc/raw/eu_sanctions/
```

## 5️⃣ OpenSanctions — aggregated PEPs + sanctions

**What it is:** a community/commercial aggregator that joins ~150 sanctions, PEP, and watchlists into a deduplicated dataset modeled in FtM (Follow the Money) schema.

**Why we use it:** it gives us a pre-aligned schema across OFAC, EU, UN, UK HMT, and PEP datasets. We do **not** rely on it as ground truth — our entity-resolution transform still runs across the raw OFAC + EU layers — but OpenSanctions is a strong second input.

- URL (standard edition, JSON, CC-BY): `https://data.opensanctions.org/datasets/latest/default/entities.ftm.json`

```bash
curl -fsSL https://data.opensanctions.org/datasets/latest/default/entities.ftm.json \
  -o /data/raw/opensanctions/entities.ftm.json
aws s3 cp /data/raw/opensanctions/entities.ftm.json s3://acme-intel-poc/raw/opensanctions/
```

> **License caveat:** the standard edition is CC-BY. The **Enriched edition** (additional PEP/UBO sources) requires a paid commercial license — out of scope for the PoC.

## 6️⃣ Wikidata — entity enrichment

**What it is:** the structured-data sibling of Wikipedia. Each entity (`Person`, `Organization`, `Country`, …) has a Q-ID with multilingual labels, type predicates, and link properties.

**Why we use it:** OpenSanctions entries often include Wikidata Q-IDs. We pull a **subset** (only Q-IDs referenced by GDELT actors or sanctions entries) to keep the volume manageable while still enriching the `Actor` ontology object with date of birth, nationality, employer, position held, etc.

### Subset extraction (SPARQL)
```sparql
# Selects Q-IDs in our seed set
SELECT ?q ?qLabel ?dob ?country ?employer WHERE {
  VALUES ?q { wd:Q42 wd:Q937 wd:Q1142 ... }  # seed set populated by the build step
  OPTIONAL { ?q wdt:P569 ?dob }
  OPTIONAL { ?q wdt:P27 ?country }
  OPTIONAL { ?q wdt:P108 ?employer }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en". }
}
LIMIT 50000
```

The seed set is built by the `actor-seed-extractor` transform (see [`06-pipelines-y-transformaciones.md`](06-pipelines-y-transformaciones.md)).

```bash
python tools/poc-geopolitica/download_wikidata_subset.py \
  --seeds s3://acme-intel-poc/staging/actor_seeds/seeds.parquet \
  --output s3://acme-intel-poc/raw/wikidata/
```

## 7️⃣ Synthetic analyst enrichment

> Needed because no public dataset gives us realistic **case-management** fields (analyst notes, source reliability score, case status). We generate these from real underlying IDs so the demo's "open investigation case" flow has plausible data to act on.

### Synthetic schemas

#### `actor_tradecraft.parquet`
| column | type | distribution |
|---|---|---|
| `actor_id` | string | uniform sample from resolved `Actor` PKs |
| `reliability_score` | float [0,1] | Beta(2, 5) (skewed low — most sources are noisy) |
| `analyst_owner` | string | sample from 12 demo analysts |
| `tradecraft_tags` | list<string> | from a fixed vocabulary {procurement, finance, logistics, propaganda, …} |
| `last_reviewed_at` | timestamp | uniform over last 12 months |

#### `case_seed.parquet`
| column | type | notes |
|---|---|---|
| `case_id` | string | UUID |
| `subject_actor_id` | string | FK |
| `status` | enum | OPEN / IN_REVIEW / CLOSED_NO_ACTION / ESCALATED |
| `priority` | enum | LOW / MEDIUM / HIGH / CRITICAL |
| `opened_at` | timestamp | |
| `assigned_to_user_id` | string | Sofía or Marcos for demo |

### Generator (deterministic, seed = 42)
```bash
python tools/poc-geopolitica/generate_tradecraft.py \
  --actor-source s3://acme-intel-poc/curated/actor/ \
  --output       s3://acme-intel-poc/raw/synthetic/ \
  --seed 42
```

### Intentional patterns in the synthetic data (so the demo is striking)
- **Planted case:** an OFAC-listed `Person` with `tradecraft_tags = [procurement, logistics]` has 3 ACLED events + 17 GDELT mentions in a single conflict-zone country in the last 14 days. This is the **anchor of UC-2** and the AIP brief.
- **Watchlist hit pattern:** 4 Wikidata-enriched `Organization`s that are not yet OFAC-listed but co-occur with an OFAC-listed person in GDELT in the same week — the candidates for UC-6 (branch demo: "extend OFAC list with 12 candidates").

> **Reproducibility:** the generator must be deterministic (fixed seed) so rehearsals match the demo exactly.

## 🗂️ Final layout in the data lake

```
s3://acme-intel-poc/
├── raw/
│   ├── gdelt/
│   │   ├── events/      year=YYYY/month=MM/day=DD/*.parquet
│   │   └── gkg/         year=YYYY/month=MM/day=DD/*.parquet
│   ├── acled/           year=YYYY/month=MM/*.parquet
│   ├── ofac/            sdn_advanced.xml (versioned by build)
│   ├── eu_sanctions/    full.xml         (versioned by build)
│   ├── opensanctions/   entities.ftm.json (versioned by build)
│   ├── wikidata/        subset.parquet
│   └── synthetic/       actor_tradecraft/, case_seed/
├── staging/    (clean, typed, deduplicated — bronze→silver output)
├── curated/    (ontology-backed tables — silver→gold)
│   ├── actor/          (post entity resolution)
│   ├── event/
│   ├── location/
│   ├── sanctions_entry/
│   ├── news_article/
│   └── link/           (link tables per link type)
└── ontology/   (materialized views served by ontology-query-service)
```

Format: **Apache Iceberg** over Parquet, partitioned by day (events) or by build id (sanctions snapshots). Iceberg REST catalog served by `iceberg-catalog-service` (Foundry-flavor admin surface over Lakekeeper).

## 🛂 Markings applied at ingest

This is the headline differentiator for this PoC. Each raw dataset is tagged with one or more markings on ingestion; downstream curated datasets inherit them unless a transform explicitly redacts.

| Dataset | Markings | Effect |
|---|---|---|
| `raw.gdelt.events` | `OPEN-SOURCE`, `ANALYST-CORE` | Sofía sees it raw; Marcos sees `url` field redacted. |
| `raw.gdelt.gkg`    | `OPEN-SOURCE`, `ANALYST-CORE` | Same as above. |
| `raw.acled`        | `OPEN-SOURCE` | Visible to both personas. |
| `raw.ofac_sdn`     | `OPEN-SOURCE`, `SANCTIONS-LIST`, `COMPLIANCE-CORE` | Marcos sees full detail; Sofía sees a degraded view (no internal program codes). |
| `raw.eu_sanctions` | Same as OFAC | |
| `raw.opensanctions`| `OPEN-SOURCE`, `SANCTIONS-LIST` | Visible to both. |
| `raw.wikidata`     | `OPEN-SOURCE` | Visible to both. |
| `raw.synthetic.*`  | `ANALYST-CORE` (tradecraft), `COMPLIANCE-CORE` (case_seed) | Persona-specific. |

> The marking enforcement is provided by `authorization-policy-service` and applied by `dataset-versioning-service` on read. See [`10-seguridad-y-gobierno.md`](10-seguridad-y-gobierno.md) for the policy definitions.
