## 1️⃣ OpenSky Network — historical and live ADS-B

**What it is:** a collaborative network that collects ADS-B messages from civil aircraft worldwide. Each `state_vector` contains `icao24, callsign, origin_country, time_position, lon, lat, baro_altitude, on_ground, velocity, heading, vertical_rate, geo_altitude, squawk, ...`.

**Why it's good for the PoC:** real, verifiable data (the customer can cross-check with FlightRadar24 on their phone), serves both *streaming* (live API) and *batch* (Trino/Impala historical) needs.

### A) Historical (batch) — via Trino (successor to Impala)
- URL: `https://opensky-network.org/data/impala`
- Access: free account at `opensky-network.org` → request access to the Trino endpoint (`trino.opensky-network.org`).
- Main table: `state_vectors_data4` — **~50 GB/month compressed in Parquet**.

#### Query template for download (12 months)
```sql
SELECT *
FROM state_vectors_data4
WHERE hour >= UNIX_TIMESTAMP('2025-04-01 00:00:00')
  AND hour <  UNIX_TIMESTAMP('2026-04-01 00:00:00')
  -- Opcional: limitar a una bbox para reducir
  AND lat BETWEEN 24 AND 72
  AND lon BETWEEN -130 AND 40
```

#### Download command (example, run on PoC day)
```bash
# Cliente Trino oficial
trino --server https://trino.opensky-network.org \
      --user $OPENSKY_USER --password \
      --execute "$(cat opensky_query.sql)" \
      --output-format CSV_HEADER > opensky_2025-2026.csv

# Convertir a Parquet particionado por día
python tools/csv_to_parquet.py \
  --input opensky_2025-2026.csv \
  --output s3://acme-poc/raw/opensky/ \
  --partition-by day
```

### B) Live (streaming) — during the demo
- REST: `GET https://opensky-network.org/api/states/all` every 5 s (authenticated: 1 req / 5 s).
- Dedicated connector in `connector-management-service` with type `opensky-rest`.

#### Connector configuration (JSON for `connector-management-service`)
```json
{
  "name": "opensky-live",
  "type": "rest-poller",
  "endpoint": "https://opensky-network.org/api/states/all",
  "auth": { "type": "basic", "user_secret": "OPENSKY_USER", "pass_secret": "OPENSKY_PASS" },
  "poll_interval_seconds": 5,
  "sink": { "type": "kafka", "topic": "opensky.states.live" },
  "schema_inference": true
}
```

### Mandatory attribution
> *"Data from the OpenSky Network — https://opensky-network.org"*. Shown in the dashboard footer.

## 2️⃣ NOAA HRRR + GFS — weather

**What it is:** NOAA numerical models. **HRRR** (High-Resolution Rapid Refresh) covers CONUS at 3 km hourly. **GFS** (Global Forecast System) covers the whole planet at 0.25°.

**Public S3 bucket (no egress fee if we are in us-east-1):**
- `s3://noaa-hrrr-bdp-pds/`
- `s3://noaa-gfs-bdp-pds/`

### Selective download command (6 months CONUS, key variables)
```bash
# Variables relevantes para aviación: viento, visibilidad, techo de nubes, turbulencia
VARS="UGRD|VGRD|VIS|HGT_ceiling|TURB"

aws s3 sync s3://noaa-hrrr-bdp-pds/hrrr.20251101 \
            s3://acme-poc/raw/noaa-hrrr/2025-11-01/ \
            --no-sign-request \
            --exclude "*" --include "*.wrfsfcf00.grib2" --include "*.wrfsfcf06.grib2"

# Post-proceso: GRIB2 → Parquet con xarray + cfgrib
python tools/grib_to_parquet.py \
  --input s3://acme-poc/raw/noaa-hrrr/ \
  --output s3://acme-poc/raw/noaa-hrrr-parquet/ \
  --vars "$VARS"
```

### How it lands in the ontology
- `weather_observation` table (granularity: nearest station to the airport + hour).
- Joins with `Flight` by `(scheduled_departure_airport, scheduled_departure_hour)` and `(arrival_airport, arrival_hour)`.

## 3️⃣ BTS On-Time Performance

**What it is:** Bureau of Transportation Statistics. Monthly CSV with every US domestic flight: `Year, Month, DayOfMonth, FlightDate, Reporting_Airline, Tail_Number, Origin, Dest, CRSDepTime, DepTime, DepDelay, ArrDelay, CarrierDelay, WeatherDelay, NASDelay, SecurityDelay, LateAircraftDelay, Cancelled, Diverted, AirTime, Distance, ...`.

**URL:** `https://www.transtats.bts.gov/PREZIP/On_Time_Reporting_Carrier_On_Time_Performance_1987_present_<YYYY>_<MM>.zip`

### Bulk download script
```bash
for year in $(seq 2018 2024); do
  for month in $(seq -w 1 12); do
    url="https://www.transtats.bts.gov/PREZIP/On_Time_Reporting_Carrier_On_Time_Performance_1987_present_${year}_${month}.zip"
    curl -fsSL "$url" -o "/data/raw/bts/${year}_${month}.zip" || echo "skip ${year}-${month}"
  done
done

# Unzip + concat + a Parquet particionado por (year, month)
python tools/bts_to_parquet.py --input /data/raw/bts/ --output s3://acme-poc/raw/bts/
```

## 4️⃣ FAA Aircraft Registry + OurAirports

### FAA Aircraft Registry
- URL: `https://registry.faa.gov/database/ReleasableAircraft.zip`
- Contains `MASTER.txt` (registrations) and `ACFTREF.txt` (models).
- Provides: `tail_number → manufacturer, model, year, engines, owner_type`.

### OurAirports
- URL: `https://davidmegginson.github.io/ourairports-data/`
- Files: `airports.csv`, `runways.csv`, `navaids.csv`, `countries.csv`.
- Provides: 80,000 airports with coordinates, IATA/ICAO, elevation.

```bash
curl -L https://davidmegginson.github.io/ourairports-data/airports.csv -o airports.csv
curl -L https://davidmegginson.github.io/ourairports-data/runways.csv  -o runways.csv
```

## 6️⃣ Synthetic MRO — generated by us

> Needed because **no public aircraft maintenance dataset is broad enough**. We generate it from the real `tail_number` values from the FAA Registry so it stays consistent.

### Synthetic schema

#### `work_orders.parquet`
| column | type | distribution |
|---|---|---|
| `wo_id` | string | UUID |
| `tail_number` | string | uniform sample from FAA Registry |
| `aircraft_model` | string | derived from FAA |
| `created_at` | timestamp | uniform over 24 months |
| `defect_code` | enum | catalog of ATA chapters (21–80) |
| `severity` | enum | LOW/MEDIUM/HIGH/CRITICAL (Pareto 70/20/8/2) |
| `discovered_during` | enum | LINE_CHECK / A_CHECK / C_CHECK / IN_FLIGHT |
| `closed_at` | timestamp nullable | NULL in ~5% (open) |
| `mttr_hours` | float | LogNormal(μ=2.5, σ=1.2) |
| `parts_used` | list<string> | 0–5 part_ids |
| `assigned_engineer_id` | string | sample from 200 engineers |

#### `parts.parquet`
| column | type | notes |
|---|---|---|
| `part_id` | string | catalog of 50,000 |
| `part_name` | string | ATA families |
| `manufacturer` | string | real sample (Honeywell, Safran…) |
| `unit_cost_usd` | float | LogNormal |
| `lead_time_days` | int | Poisson |
| `compatible_models` | list<string> | |

#### `inspections.parquet`
| column | type |
|---|---|
| `inspection_id`, `tail_number`, `inspection_type` (A/B/C/D check), `scheduled_at`, `performed_at`, `findings_count` |

### Generator (to be implemented when the PoC runs)
```bash
# Pseudocódigo del generador
python tools/generate_mro.py \
  --tail-numbers from-faa-registry \
  --start 2024-01-01 --end 2026-04-01 \
  --output s3://acme-poc/raw/mro/ \
  --target-rows 250_000_000 \
  --seed 42
```

> **Important:** the generator must be **deterministic (fixed seed)** so that rehearsals and the demo are reproducible.

### Intentional patterns in the synthetic data (so the demo is spectacular)
- **Planted recurring defect:** `defect_code = 27-30 (Flight Controls — Aileron)` with anomalous frequency in the A320 fleet between `2026-02` and `2026-04`. This makes UC-3 *"recurring defect detection"* visually striking.
- **Weather–delay correlation:** the synthetic MRO delays correlate with real HRRR storms, so the copilot can explain them.
- **Part with high lead time:** a specific part (`part_id = HW-AIL-7421`) has a 21-day lead time — used in the MRO workflow.

## 🗂️ Final layout in the data lake

```
s3://acme-poc/
├── raw/
│   ├── opensky/           year=YYYY/month=MM/day=DD/*.parquet
│   ├── noaa-hrrr/         year=YYYY/month=MM/day=DD/hour=HH/*.parquet
│   ├── bts/               year=YYYY/month=MM/*.parquet
│   ├── faa-registry/      master.parquet, acftref.parquet
│   ├── ourairports/       airports.parquet, runways.parquet
│   └── mro/               work_orders/, parts/, inspections/
├── staging/    (limpio, tipado, deduplicado — output de pipelines bronze→silver)
├── curated/    (tablas de la ontología — silver→gold)
└── ontology/   (vistas materializadas servidas por ontology-query-service)
```

Format: **Apache Iceberg** over Parquet, partitioned by date. Iceberg REST catalog served by `iceberg-catalog-service` (Foundry-flavor admin surface over Lakekeeper).
